Vague support requests create free project management.

Post-launch support gets messy when every bug, edit, password issue, plugin warning, and “quick change” lands in a different thread. A support request form makes the client slow down enough to describe the problem and gives you a clean way to route the request.

The goal is not to hide from clients. The goal is to separate included bug fixes from new work before your afternoon disappears.

What the form should capture

Actual issue details

Ask for the affected URL, what happened, what they expected, when it started, and whether it can be reproduced.

Evidence

Require screenshots, screen recordings, error messages, device/browser info, and login/access notes when relevant.

Urgency without drama

Let clients mark business impact, but define what counts as urgent versus normal queue work.

Scope routing

Include a line that bug fixes inside the agreed warranty window are handled differently from new features, content edits, or third-party platform work.

Client confirmation: [typed name]

Support intake checklist

  • ☐ Put the form link in the handoff packet and support policy.
  • ☐ Define what counts as urgent before launch.
  • ☐ Ask for screenshots or recordings by default.
  • ☐ Tell clients not to paste passwords into the form.
  • ☐ Add one scope-check checkbox before submission.
  • ☐ Route billable work to a change-request form or care-plan quote.
  • ☐ Save the completed request with the project record.

Boundary line to include

Please use this form for post-launch website support so we can collect the details needed to troubleshoot quickly. If the request is outside the included support window, original project scope, or active care plan, I’ll confirm the estimate before starting billable work.
