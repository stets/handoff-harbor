An Elementor site is not handed off just because WordPress is live.

The messy version: the client can log in but does not know what not to touch, Elementor Pro renews on the freelancer's account, forms still route to the wrong inbox, backups are assumed but not verified, plugins update without a plan, and every layout change becomes unpaid troubleshooting.

This is not legal, tax, or WordPress/Elementor policy advice. It is a practical closeout checklist for freelancers and small studios so the client receives ownership, access, editing rules, acceptance, and support boundaries in writing.

What to document before an Elementor project is accepted

WordPress, hosting, and admin ownership

Confirm the client has the right WordPress admin role, hosting access, registrar/DNS ownership, backup access, recovery email, 2FA expectations, and a list of any remaining agency accounts.

Elementor Pro, theme, and plugin licenses

Record who owns the Elementor Pro license, theme license, premium plugin subscriptions, renewal dates, update responsibility, and what happens if a license is removed or not renewed.

Forms, templates, global styles, and custom code

List Elementor forms, notification recipients, CRM/newsletter routes, global colors/fonts, templates, popups, custom CSS/JS, tracking snippets, and anything the client should not edit casually.

Training and support boundaries

Explain safe edits versus risky layout/template/plugin work, included bug-fix window, excluded new work, change-request process, and the maintenance/care-plan option.

Copy/paste Elementor handoff checklist

  • ☐ WordPress admin owner, client users, roles, recovery email, 2FA method, and contractor cleanup are confirmed.
  • ☐ Hosting, registrar, DNS, SSL, CDN/cache, email records, staging site, and deployment notes are documented.
  • ☐ Elementor Pro license owner, renewal date, connected sites, plan limits, theme license, and premium plugin licenses are listed.
  • ☐ Theme, child theme, Elementor global styles, templates, headers/footers, popups, custom CSS/JS, and reusable sections are documented.
  • ☐ Forms, notification emails, SMTP/mail deliverability, spam settings, CRM/newsletter/booking/payment integrations, and test submissions are verified.
  • ☐ Analytics, pixels, Search Console, SEO plugin settings, schema, redirects, cookie/privacy tools, and conversion events are captured.
  • ☐ Backup system, restore owner, retention window, update process, plugin/theme conflict plan, and maintenance responsibility are explained.
  • ☐ Client editing notes explain safe text/image edits, risky Elementor layout/template changes, plugin settings, custom code, cache clearing, and media sizing.
  • ☐ Known issues, accepted tradeoffs, future recommendations, and unsupported requests are written down before final acceptance.
  • ☐ Included support window is stated, and new pages, layout work, plugin issues, SEO, copy, integrations, troubleshooting, or ongoing updates route to paid work.

Before we call the project accepted, I want to make sure the operating pieces are clear: WordPress/admin access, hosting, domain/DNS/SSL, Elementor Pro and plugin licenses, forms, analytics/SEO, backups, editing notes, and the support window. Please review the handoff notes below by [date]. Launch bugs tied to the original scope are covered through [support end date]. New pages, layout changes, plugin issues, forms, integrations, SEO, copy changes, troubleshooting, or ongoing maintenance can be handled through a separate change request or care plan.
