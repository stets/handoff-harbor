Backups are boring until the client needs one.

The risky version of handoff is “your host takes backups somewhere, probably.” The cleaner version documents where backups live, who can access them, how often they run, what they include, how long they are retained, and who is responsible if the site breaks after launch.

This is especially important for WordPress, Shopify theme work, Webflow exports, custom forms, membership sites, and anything with client-owned content or transactions.

What to document before closeout

Backup source

List whether backups come from hosting snapshots, a WordPress plugin, Webflow export, database dump, Git repo, or manual file archive.

Restore owner

Name who is responsible for restoring the site after handoff: client, host, maintenance provider, or you under a paid care plan.

Retention window

Capture backup frequency, retention length, storage location, and whether offsite backups exist.

Restore proof

Run or confirm at least one restore path before claiming the client is protected.

Copy/paste website backup handoff checklist

  • ☐ Identify the primary backup method: host snapshot, plugin, export, repo, database dump, or manual archive.
  • ☐ Confirm what the backup includes: files, uploads, database, theme, plugins, forms, products, orders, members, and custom code.
  • ☐ Confirm backup schedule and retention window.
  • ☐ Confirm storage location and who has access.
  • ☐ Save emergency restore contact details for the host or maintenance provider.
  • ☐ Test or verify a restore path before final handoff.
  • ☐ Document anything backups do not cover, such as email, DNS, third-party app settings, paid plugin licenses, or external automations.
  • ☐ Put renewal costs, plugin licenses, hosting plan, and backup tool billing in the handoff packet.
  • ☐ Explain who handles restores after launch and what counts as paid support.
  • ☐ Send the backup note before final payment or maintenance plan decision.

Client backup handoff note

Backup method: [hosting snapshots / WordPress plugin / Webflow export / manual archive / other] Backup frequency: [daily / weekly / manual] Retention window: [7 days / 30 days / other] Storage location: [hosting dashboard / cloud folder / plugin storage / repo] Restore access: [who can restore the site] Emergency contact: [host support URL / maintenance provider / internal owner] Last restore test: [date or not tested] What the backup includes: - [site files] - [database/content] - [uploads/media] - [theme/custom code] - [forms/products/orders/members if relevant] What the backup does not include: - [email/DNS/third-party apps/licenses/automations] After handoff, [client/company] is responsible for maintaining backups unless a paid care plan is active. Restore help after launch is [included/not included] and will be handled as [support terms].

Quick restore questions to answer

  • ☐ If the site breaks tomorrow, who logs in first?
  • ☐ Can the client access the host or backup dashboard?
  • ☐ Is there a backup from before launch changes?
  • ☐ Are database and media files both covered?
  • ☐ Are paid licenses and plugin updates still active?
  • ☐ Is restore work covered by the project, a warranty window, or a paid maintenance plan?
